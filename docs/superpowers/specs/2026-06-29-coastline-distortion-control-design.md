# Controllable coastline distortion

## Problem

The map background is generated by feeding an edit model a flat land/sea **control
image** (green land / blue sea) plus a style reference. The control image is built from
the board's tile layout, then its coastline is **domain-warped** through a smooth noise
vector field so hex edges bend into natural shores (`renderLandMask` in
`packages/terrain/src/masks.ts`).

The warp magnitude is `base.coastWarp.amplitude` — the max pixel displacement. It is
currently **160px** on a 1024px-wide control image (~16% of width). That is too loose:
the background's land/sea boundary wanders far from the actual tiles, yet the background's
whole job is to let players distinguish land tiles from sea tiles, so the layout must hug
the tile layout closely.

Two needs:
1. A **much lower default** so the coastline stays aligned to the tiles out of the box.
2. A way to **control the distortion amount** without hand-editing the profile JSON, so
   amplitude can be swept cheaply (the control render is fal-free — no API key, no cost).

## Decisions

- Mechanism: **CLI flag override** on the `gen:map-control` command (not a normalized
  0–1 dial; raw pixels match the existing schema).
- New default amplitude: **30px** (~3% of 1024px width) — coastline barely leaves the
  tile boundary, mostly just softening hex corners.
- Flag scope: **amplitude only**. `scale` and `seed` stay profile-only.

## Design

### 1. Lower the default (two places, kept in sync)

- `packages/terrain/profiles/map.json`: `base.coastWarp.amplitude` **160 → 30**.
- `packages/terrain/src/mapProfile.ts`: zod `amplitude` default **160 → 30**, and update
  the doc comment that currently cites 160.

### 2. CLI amplitude override

`packages/terrain/src/mapControlCli.ts`:

- New usage:
  `pnpm --filter @sengoku-jidai/terrain gen:map-control <mapId> [--amplitude <px>]`
- Parse argv with `node:util` `parseArgs`. When `--amplitude` is provided it overrides
  `base.coastWarp.amplitude`; otherwise the profile value is used.
- Validation: must parse to a finite number ≥ 0. Reject non-numeric / negative values
  with a clear error. `0` is valid and disables the warp.
- Log the **effective** amplitude and its source so a sweep is self-documenting, e.g.
  `[terrain] coastWarp amplitude: 30 (profile)` or `… 50 (override)`.

### 3. Testability

Extract a pure helper so `main()` stays thin and the parsing/validation is unit-tested:

- New module `packages/terrain/src/mapControlArgs.ts` exporting
  `parseMapControlArgs(argv: string[]): { mapId: string; amplitude?: number }`.
  - `argv` is the post-`node script` args (i.e. `process.argv.slice(2)`).
  - Missing mapId → throws with the usage string.
  - `--amplitude` non-numeric or `< 0` → throws a clear error.
  - No `--amplitude` → `amplitude` is `undefined`.
- New test `packages/terrain/test/mapControlArgs.test.ts` covering: valid override,
  no flag (undefined), `0` allowed, invalid (`foo`) throws, negative throws, missing
  mapId throws.
- `mapControlCli.ts` calls `parseMapControlArgs(process.argv.slice(2))` and applies the
  override onto `base.coastWarp.amplitude` before calling `renderLandMask`.

### 4. Scope guard (YAGNI)

No changes to `masks.ts` warp logic, the web app, or any other CLI
(`gen`, `gen:base`, `gen:matrix`, `gen:map`).

## Verification

- `pnpm --filter @sengoku-jidai/terrain test` (new arg-parse test passes).
- `pnpm --filter @sengoku-jidai/terrain typecheck`.
- Manual: `gen:map-control <mapId>` writes a control with a tight coastline; same with
  `--amplitude 80` produces a looser one; `--amplitude -1` and `--amplitude foo` error.
  (Visual confirmation of the rendered control is left to the user / CI per the
  no-local-browser-verification constraint.)
